# Stateful Tool Environments for Oumi Synthesis

## Overview

Extends the existing agentic tool synthesis system (`ani/agentic-synth`) with **stateful environments**. Today, tools produce independent outputs per call (DETERMINISTIC or GENERATED). This design adds environments—stateful containers that tools operate on—so that tool results are consistent with a shared, evolving state across a conversation.

## Problem

Current tool synthesis is stateless. Each tool call gets an independent result:
- DETERMINISTIC: pre-sampled canned output, same for every call to that tool
- GENERATED: LLM-simulated output conditioned on conversation history

This can't produce realistic multi-tool trajectories where state matters. For example, `create_file("notes.txt")` followed by `read_file("notes.txt")` should return the content that was just created. Today, the GENERATED simulator has no memory of the create call's effect.

## Design

### New Types

#### `EnvironmentAttribute` (in `tool_params.py`)

Config-level definition of an environment:

```python
@dataclass
class EnvironmentAttribute:
    id: str                                    # unique identifier
    name: str                                  # display name
    description: str                           # what this environment represents
    system_prompt: str                         # instructions for the LLM managing state
    state_schema: dict[str, Any] | None = None # JSON Schema for state; None = auto-generate
    initial_state: dict[str, Any] | None = None # starting state; None = auto-generate
```

- If `state_schema` is `None`, it is generated by an LLM at initialization time from the bound tools' schemas and descriptions.
- If `initial_state` is `None`, it is generated by an LLM from the schema.
- Both are validated: schema must be valid JSON Schema, initial state must conform to schema.

#### `ToolOutputStrategy.ENVIRONMENT`

A new enum value added to `ToolOutputStrategy`:

```python
class ToolOutputStrategy(str, Enum):
    DETERMINISTIC = "deterministic"
    GENERATED = "generated"
    ENVIRONMENT = "environment"    # NEW — output comes from a stateful environment
```

This avoids the existing `__post_init__` validation on `ToolAttribute` that requires `deterministic_outputs` or `generated_output` depending on strategy. Environment-bound tools use `output_strategy=ENVIRONMENT` and skip those checks.

#### Tool-to-Environment Binding

Two new optional fields on the existing `ToolAttribute`:

```python
@dataclass
class ToolAttribute:
    # ... existing fields (id, name, description, output_strategy, parameters,
    #     output_schema, deterministic_outputs, generated_output) ...

    environment: str | None = None  # references EnvironmentAttribute.id; None = stateless
    read_only: bool = True          # does this tool modify its environment's state?
```

- Tools with `environment=None` continue using the existing DETERMINISTIC/GENERATED path unchanged.
- Tools with `environment` set must use `output_strategy=ENVIRONMENT`. Validation in `ToolAttribute.__post_init__` enforces this: if `environment` is set, `output_strategy` must be `ENVIRONMENT`; if `output_strategy` is `ENVIRONMENT`, `environment` must be set.
- The relationship is many-to-one: many tools can share one environment, but each tool binds to at most one environment.

### `GeneratedEnvironment` Runtime Class

New file: `src/oumi/core/synthesis/environment.py`

```python
class GeneratedEnvironment:
    def __init__(self, config: EnvironmentAttribute, inference_engine: BaseInferenceEngine):
        self._config = config
        self._inference_engine = inference_engine
        self._state: dict[str, Any] | None = deepcopy(config.initial_state)
        self._state_schema: dict[str, Any] | None = config.state_schema

    def initialize(self, tools: list[ToolAttribute]) -> None:
        """Generate state_schema and/or initial_state if not provided.

        Called once at synthesis start, before any conversations.
        Handles three cases:
        - Both None: generate schema from tools, then generate initial state from schema.
        - schema provided, initial_state None: generate initial state from schema.
        - Both provided: validate initial_state against schema, no generation needed.

        Uses LLM to generate schema from tool definitions, then
        generates initial state from schema. Validates both.
        """

    def step(self, tool: ToolAttribute, arguments: dict[str, Any]) -> str:
        """Execute a tool call against this environment.

        Uses tool.read_only to determine whether state is updated.

        1. _generate_result: LLM sees (system_prompt, state, tool_call) -> result
        2. _update_state: LLM sees (system_prompt, state_schema, state, tool_call, result) -> new_state
           Skipped if tool.read_only is True.
        3. _validate_state: jsonschema.validate(new_state, state_schema)
           On failure: retry _update_state once with error context. On second failure: log warning, keep old state.
        """

    def reset(self) -> None:
        """Reset to initial state. Called between conversations."""
```

**Per-sample isolation**: Each sample in a batch gets its own `deepcopy` of the `GeneratedEnvironment` so that state does not leak between conversations. The `reset()` method is a convenience for resetting within a single sample if needed.

**Cost note**: Each non-read-only tool call requires two sequential LLM inferences (`_generate_result` + `_update_state`), plus a potential retry on validation failure. For conversations with many mutating tool calls, this adds significant LLM cost. A future optimization could consolidate both into a single LLM call that returns both the result and the updated state.

#### LLM Prompt Design

**Schema generation** (`initialize`, when `state_schema` is None):
- Input: environment description, system prompt, all bound tools with their parameter schemas, output schemas, descriptions, and read_only flags
- Instruction: design a JSON Schema for the state. Every field any tool reads or writes must be represented.
- Validation: parsed output must be valid JSON Schema.

**Initial state generation** (`initialize`, when `initial_state` is None):
- Input: environment description, validated schema
- Instruction: generate a realistic initial state conforming to the schema.
- Validation: `jsonschema.validate(initial_state, state_schema)`. Retry up to 2 times on failure.

**Result generation** (`_generate_result`):
- System: `{system_prompt}` + current state as JSON
- User: tool name, arguments, instruction to produce output as JSON matching tool's output schema

**State update** (`_update_state`):
- System: `{system_prompt}` + state schema definition
- User: current state, tool call, tool result, instruction to update state as JSON conforming to schema

### Integration Points

#### `ToolExecutor` Changes

New constructor parameters:

```python
class ToolExecutor:
    def __init__(self, tools: list[ToolAttribute], environments: list[GeneratedEnvironment] | None = None):
        # ... existing tool lookups ...
        self._environments: dict[str, GeneratedEnvironment] = {
            env._config.id: env for env in (environments or [])
        }
```

New method:

```python
def resolve_output_with_env(self, tool_call: dict[str, Any]) -> str | None:
    """Route tool call through its environment if bound.

    Looks up the tool by name, checks if it has an environment binding.
    If yes, calls env.step(tool, arguments) and returns the result string.
    If no (stateless tool), returns None so the caller falls through to
    the existing DETERMINISTIC/GENERATED path.
    """
```

The existing `resolve_output` and `build_generated_simulator_prompt` methods remain unchanged.

#### `ConversationSynthesizer` Changes

The tool-use loop in `_synthesize_all_samples` gains one branch, checked **before** the existing `resolve_output` call:

```python
# NEW: check environment binding first
env_result = tool_executor.resolve_output_with_env(tool_call)
if env_result is not None:
    result = env_result
else:
    # existing path: DETERMINISTIC or GENERATED
    result = tool_executor.resolve_output(tool_call, deterministic_selections[idx])
    if result is None:
        # GENERATED: build simulator prompt, queue for batch inference
        ...
```

**Important**: Environment-bound tool calls within a conversation must be processed **sequentially** (each `step()` may mutate state that the next call depends on). They cannot be batched like GENERATED simulator calls. This is already the case in the current loop structure — tool calls within a single assistant turn are processed one at a time.

**Lifecycle**:
- In `synthesize()`, before `_synthesize_all_samples`: create `GeneratedEnvironment` instances from config, call `initialize(bound_tools)` on each to generate schema/initial_state if needed.
- In `_synthesize_all_samples`, for each sample: `deepcopy` all environments so each sample has isolated state.
- During a sample's conversation: `step()` is called per tool call, state accumulates within that sample.

#### `GeneralSynthesisParams` Changes

One new field:

```python
@dataclass
class GeneralSynthesisParams:
    # ... existing fields ...
    environments: list[EnvironmentAttribute] | None = None
```

New validation in `__post_init__`:
- Environment ids are unique
- Every `tool.environment` references a valid environment id
- Tools with `environment` set don't require `output_strategy`, `generated_output`, or `deterministic_outputs`

### Error Handling

Three error types from the design doc:

| Type | Source | Handling |
|------|--------|----------|
| Arg errors | LLM produces bad arguments | Existing `ToolExecutor.validate_arguments()` — unchanged |
| Env-derived | Valid call, invalid given state (e.g. read after delete) | Environment LLM produces error result naturally from state |
| Random errors | Transient failures | Future: configurable `failure_rate` per tool (not yet implemented) |

### What Does NOT Change

- `Role.TOOL` and `FinishReason.TOOL_CALLS` — already defined
- `ToolOutputStrategy` enum — DETERMINISTIC and GENERATED remain as-is (ENVIRONMENT is added)
- `DeterministicToolOutput` and `GeneratedToolOutput` — unchanged
- `ToolExecutor.parse_tool_call()` — unchanged
- `ToolExecutor.validate_arguments()` — unchanged
- `ToolExecutor.sample_deterministic_outputs()` — unchanged
- `ToolExecutor.build_tool_catalog()` — unchanged (env-bound tools appear in catalog the same way)
- `ToolExecutor.format_tool_call_message()` / `format_tool_result_message()` — unchanged
- `MultiTurnAttribute.available_tools` — references tool ids regardless of whether tools are stateless or env-bound
- All output formatting (OpenAI-compatible message dicts) — unchanged

### Future Extensions

- **Executable environments**: A subclass of `GeneratedEnvironment` that runs real code (Python dict as filesystem, SQLite as database). Same `step()` interface, deterministic results.
- **Registry integration**: If tools/environments become reusable Python classes across configs, add `RegistryType.TOOL`, `RegistryType.ENVIRONMENT` and registration decorators.
- **Many-to-many bindings**: If a tool genuinely needs to operate on multiple environments atomically, extract bindings into a separate `ToolEnvironmentMapping` config object.
