# Oumi Onboarding Flow - Detailed Analysis

## Overview

The Oumi onboarding flow is a CLI-based system designed to help users configure and generate training/evaluation configurations from their data. It supports both interactive (wizard) and non-interactive modes.

## Code Locations

- **CLI Entry Point**: `src/oumi/cli/onboard.py` (~5,600+ lines)
- **Core Module**: `src/oumi/onboarding/`
  - `config_builder.py` - Builds Oumi configs from analyzed data
  - `data_analyzer.py` - Analyzes data files and extracts schema
  - `field_mapper.py` - Maps customer columns to Oumi placeholders
  - `llm_analyzer.py` - Uses LLM to infer domain context

## CLI Commands

Access via: `oumi onboard <command>`

| Command | Purpose |
|---------|---------|
| `wizard` | Interactive 4-step wizard |
| `generate` | Non-interactive config generation |
| `templates` | List available templates |
| `analyze` | Analyze data and show suggestions |

---

## Wizard Flow (4 Steps)

### Step 1: Define Your Task

**Function**: `_wizard_step_task()`

- User describes what the model should do
- If LLM enabled: Analyzes files to suggest task description
- Generates system prompt (LLM-generated or manual)
- Infers task type: extraction, classification, generation, transformation, qa
- **Output**: `TaskSpec` with description, system_prompt, task_type, example_output

### Step 2: Input Distribution

**Function**: `_wizard_step_inputs()`

- Defines what data will be inputs to the model
- If LLM enabled: AI detects best input columns
- Otherwise: Uses fallback heuristics
- Detects input format: single_turn, multi_turn, document, structured
- **Output**: `InputSpec` with source_column, format, samples

### Step 3: Output Quality Criteria

**Function**: `_wizard_step_outputs()`

- Defines what makes a good response
- If LLM enabled: Suggests quality criteria based on task
- Comma-separated criteria list (e.g., "accurate, helpful, clear")
- Optional: Gets one example of good output
- **Output**: `OutputSpec` with criteria list and example

### Step 4: Generate Configs

**Function**: `_wizard_step_generate()`

- Creates synthesis config using `SynthConfigBuilder`
- Creates judge configs for each quality criterion (max 3)
- Saves all configs to output directory
- **Output Files**:
  - `synth_config.yaml` - For data synthesis
  - `judge_*.yaml` - For quality evaluation judges

## Command Options

### Wizard Command

```bash
oumi onboard wizard --data <path> [options]
```

| Option | Description | Default |
|--------|-------------|---------|
| `--data, -d` | Path to data file or directory | Required |
| `--output, -o` | Directory to save configs | `./oumi_configs` |
| `--llm / --no-llm` | Use LLM for AI analysis | `--no-llm` |
| `--engine, -e` | LLM engine: ANTHROPIC, OPENAI, DEEPSEEK, TOGETHER | ANTHROPIC |
| `--model, -m` | Model name | Engine default |
| `--verbose, -v` | Show detailed output | False |
| `--num-samples, -n` | Number of samples to generate | 10 |

### Supported Data File Types

- `.csv`, `.json`, `.jsonl`
- `.xlsx`, `.xls` (Excel)
- `.docx`, `.doc` (Word)

|------|---------|

1. **extraction** - Extract structured information

3. **generation** - Generate new text content
4. **transformation** - Transform from one format to another
5. **qa** - Answer questions based on context

## File Role Guidance (Multi-File Analysis)

When analyzing multiple files, the system suggests roles:

- `primary` - Main training dataset
- `reference` - Knowledge base for grounding
- `rules` - System instructions/guidelines
- `examples` - Sample input-output pairs

## Column Role Guidance

- `context` - Source content for generation
- `question` - User inputs/questions
- `answer` - Model outputs/responses
- `reference_values` - Valid option lists
- `label` - Quality labels/scores
- `metadata` - Additional context columns

## Cache System

The wizard implements caching for resumable flows.

**Cache File**: `.wizard_cache.yaml`

```yaml
task:
  description: <user task description>
  system_prompt: <auto-generated or refined prompt>

inputs:
  format: single_turn|multi_turn|document|structured
  source_column: <column name>
  samples: <example data samples>

outputs:
  criteria: [list of quality criteria]
  example: <optional example output>

files:
  - path: <file path>
    suggested_purpose: <LLM analysis>
    suggested_role: primary|reference|rules|examples
    content_hash: <hash for cache invalidation>

domain_analysis:
  domain: <identified domain>
  terminology: [list]
  quality_signals: [list]
  common_issues: [list]
  suggested_persona: <persona description>

completed_steps: [task, inputs, outputs, generate]
```

**Navigation Logic**:

1. Check for existing `.wizard_cache.yaml`
2. If found: Show cache summary with resume/edit/restart options
3. Each step checks `completed_steps` list
4. If step completed: Skip and show summary
5. Cache saved after each step

### Step-by-Step: LLM vs Fallback

#### Step 1: Task Definition

| Aspect | LLM Mode | Fallback Mode |
|--------|----------|---------------|
| **Task Description** | `_analyze_task_from_files()` analyzes file contents, column names, sample data to suggest task | Simple prompt: "What should your model do?" |
| **System Prompt** | `_generate_system_prompt()` creates domain-aware 2-4 sentence prompt; user can refine with feedback | Default template: `"You are a helpful assistant. {task_description}"` |
| **Task Type** | `_infer_task_type()` determines type from description + suggests output format | Manual menu selection from 5 types (default: "generation") |

**Task Types Available:**

1. extraction - Extract structured information
2. classification - Categorize into predefined labels
3. generation - Generate new text content
4. transformation - Transform from one format to another
5. qa - Answer questions based on context

---

#### Step 2: Input Column Detection

| Aspect | LLM Mode | Fallback Mode |
|--------|----------|---------------|
| **Column Selection** | `_detect_input_source()` - LLM analyzes all columns with samples, determines best input column(s) with reasoning | `_fallback_input_detection()` - keyword matching + first non-ID column |
| **Format Detection** | LLM classifies: single_turn, multi_turn, document, structured, instruction | Always defaults to `"single_turn"` |
| **Reasoning** | LLM provides explanation for selection | `"Selected using keyword heuristics"` |

**Fallback Heuristics in `_fallback_input_detection()`:**

```python
# Skip patterns (columns to ignore)
skip_patterns = ["id", "uuid", "_id", "created", "updated", "timestamp", "index"]

# Keyword matching (priority columns)
keywords = ["question", "input", "query", "prompt", "text", "content", "message"]
```

**Fallback Selection Logic:**

1. Look for columns matching keywords → use first match
2. If no keyword match → pick first non-ID column
3. If all columns match skip patterns → use first column
4. Last resort → column index 0

---

#### Step 3: Quality Criteria

| Aspect | LLM Mode | Fallback Mode |
|--------|----------|---------------|
| **Criteria Suggestion** | `_suggest_quality_criteria()` - LLM suggests 3-5 task-specific criteria as JSON array | Hard-coded default: `["accurate", "helpful", "clear"]` |
| **User Interaction** | User can accept or customize | Direct prompt with default shown |

**LLM Prompt for Criteria:**

```
For this task, suggest 3-5 quality criteria for evaluating responses.
Task: {task_description}
Return a JSON array like: ["accurate", "helpful", "clear", "complete"]
```

### Summary: LLM vs Fallback Decision Points

| Step | Function | LLM-Based | Fallback |
|------|----------|-----------|----------|
| Task Description | `_analyze_task_from_files()` | File analysis → suggestion | Manual input |
| System Prompt | `_generate_system_prompt()` | Domain-aware prompt | Default template |
| Task Type | `_infer_task_type()` | Inferred from description | Manual menu (default: generation) |
| Input Columns | `_detect_input_source()` | AI selects best columns | Keyword matching |
| Input Format | `_detect_input_format()` | Classifies from samples | Always "single_turn" |
| Quality Criteria | `_suggest_quality_criteria()` | Task-specific criteria | ["accurate", "helpful", "clear"] |
| File Roles | `_analyze_file_purposes()` | LLM categorizes | Defaults to "context" |
| Domain Analysis | `llm_analyzer.analyze()` | Full domain extraction | Not available |

## Architecture Summary

1. **Modular Design**: Separate analysis, inference, mapping, and building components
2. **LLM-Optional**: Full functionality without LLM; LLM enhances with domain analysis
3. **Cache-Driven**: Supports resumable, iterative onboarding
4. **Config-First**: Generates YAML configs integrating with Oumi infrastructure
5. **Rich CLI**: Tables, panels, and progress indicators using Rich library
